# 贡献指南 (Contributing)

感谢你关注并参与本项目。

本指南用于帮助贡献者以一致、可复现、可维护的方式提交改动。

## 贡献方式

你可以通过以下方式参与：

- 提交 Issue（Bug、改进建议、文档问题）
- 提交 Pull Request（代码修复、功能优化、文档补充）
- 改进测试用例、示例数据说明、打包文档

## 开始之前

1. 先阅读项目说明: [README.md](README.md)
2. 确认本地 Python 版本满足要求
3. 安装依赖：

```bash
pip install -r requirements.txt
```

## 分支与开发流程

建议使用如下流程：

1. Fork 仓库并克隆到本地
2. 从 `main` 创建功能分支
3. 在分支中完成开发与自测
4. 提交 Pull Request 到 `main`

示例：

```bash
git checkout -b feat/your-feature-name
```

## 提交规范

建议使用清晰、可检索的提交信息：

- `feat:` 新功能
- `fix:` Bug 修复
- `refactor:` 重构（不改变外部行为）
- `docs:` 文档更新
- `test:` 测试相关
- `chore:` 构建/工具链/杂项

示例：

```text
fix: refresh preprocess filter columns after data load
```

## Pull Request 要求

提交 PR 前请确保：

1. 改动目标明确，范围尽量小
2. 不包含无关重构或大面积格式化
3. 本地可运行，核心流程不回归
4. 更新必要文档（README、打包说明、注释）

PR 描述建议包含：

- 改动背景
- 主要改动点
- 验证步骤
- 风险与兼容性说明
- 截图（如涉及 UI）

## 代码风格与约定

- 保持现有项目结构与命名风格
- 优先最小改动原则，避免无关文件变更
- 注释仅在必要处补充，说明“为什么”而非“做了什么”
- 新增依赖需同步更新 [requirements.txt](requirements.txt)

## 回归验证建议

提交前建议至少执行：

1. 语法检查：

```bash
python -m compileall .
```

2. 手工核心流程验证：

- 数据加载
- 特征/目标选择
- 运行分析
- 可视化生成
- 导出结果

## Issue 提交建议

请尽量提供：

- 运行环境（OS、Python 版本）
- 复现步骤
- 期望行为与实际行为
- 错误日志或截图

## 安全与合规

- 请勿提交敏感数据、隐私信息或密钥
- 如发现潜在安全问题，请不要公开披露细节，可通过私下渠道联系维护者

## 行为准则

请保持尊重、专业、建设性的沟通方式。

我们欢迎不同背景的贡献者共同改进项目。
